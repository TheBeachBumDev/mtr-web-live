Host commands — restart one module or rebuild the stack

Run on the server from the directory that contains docker-compose.yml (for example ~/mtr-web-live). Project name -p mtr-web-live must match your deployment.

Restart a single Compose service (lightweight)

docker compose -f docker-compose.yml -p mtr-web-live restart <service>

Use the Compose service column above. Typical values: core, mtr_live, download_test, fieldtech, ipam, monitoring, location_sync, routers, backhauls, stock_management, purchase_orders, postgres, redis.

docker compose -f docker-compose.yml -p mtr-web-live restart core
docker compose -f docker-compose.yml -p mtr-web-live restart postgres

Full stack rebuild (rebuild images and recreate all containers — expect downtime)

docker compose -f docker-compose.yml -p mtr-web-live up -d --build --force-recreate

If you see 502 after a rebuild, bring anything still stopped back up:

docker compose -f docker-compose.yml -p mtr-web-live up -d

Clone Rebuild (Admin)

Creates a full target replica over SSH. Confirmation phrase: CLONE <target_host>

Target prerequisites (run once per new target host)
sudo apt-get update sudo apt-get install -y docker.io docker-compose-plugin rsync ufw curl openssl sudo systemctl enable --now docker sudo usermod -aG docker wibernetdev

Then reconnect SSH and verify: docker --version, docker compose version, rsync --version.

Firewall UI / host agent — Run on the target once after the stack comes up (from the clone dir, e.g. ~/mtr-web-live): installs the mtr-firewall-agent systemd service, opens UFW from the Docker bridge subnet to port 9191, aligns FIREWALL_AGENT_URL / token in .env.compose, and recreates core.

cd /path/to/mtr-web-live   # same folder clone restored to
bash scripts/security/bootstrap_firewall_agent.sh

Without this, /firewall shows errors like “Firewall agent unreachable” (http://172.18.0.1:9191): containers reach the host via the Docker gateway; nothing listens until the agent is installed. The clone copies DB/config only — not host UFW state.

To reuse the primary’s agent secret instead of a fresh token, copy /etc/default/mtr-firewall-agent from primary → standby before bootstrap (then skip overwriting token), or edit standby so agent env matches cloned FIREWALL_AGENT_TOKEN and reopen port 9191 for the compose subnet.

One-time run on standby (after clone / compose up)

SSH to the target, then from the restored project directory (adjust the path):

Location Sync: full clone already appends LOCATION_SYNC_SCHEDULER_ENABLED=0 so the location_sync container does not run the nightly Splynx pull on standby (cache matches the cloned DB). Promote Standby sets it back to 1.

Remote configuration requirements

Run these on the standby / target host before the first clone (or after changing OS). Clone uses non-interactive SSH and expects the same tools as below.

1. Networking & SSH

  • SSH daemon listening on the port you enter in Clone (firewall allows inbound).
  • Key-based login for the clone user (clone cannot type passwords). Test from primary: ssh -o BatchMode=yes -i /path/to/private_key -p PORT user@host 'echo KEY_OK'
  • SSH user must be able to run Docker (see below) or use root.

2. Fix broken apt cdrom repo (Ubuntu installs)

If apt-get update errors about file:/cdrom:

sudo sed -i '/^deb cdrom:/s/^/#/' /etc/apt/sources.list
sudo apt-get update

3. Docker Engine & Compose v2

Engine plus Compose v2 (docker compose). Prefer the plugin from apt; on older repos install the CLI plugin binary.

sudo apt-get install -y docker.io rsync
sudo systemctl enable --now docker
sudo usermod -aG docker YOUR_USER

Try plugin package first:

sudo apt-get install -y docker-compose-plugin
docker compose version

If docker-compose-plugin is not found, install the v2 plugin manually:

sudo mkdir -p /usr/local/lib/docker/cli-plugins
sudo curl -SL https://github.com/docker/compose/releases/download/v2.29.7/docker-compose-linux-x86_64 \
  -o /usr/local/lib/docker/cli-plugins/docker-compose
sudo chmod +x /usr/local/lib/docker/cli-plugins/docker-compose
docker compose version

Avoid legacy Python docker-compose v1 if it errors on OpenSSL — use docker compose v2 instead.

4. Daemon & permissions

sudo systemctl status docker
docker ps

If "Cannot connect to the Docker daemon", fix group membership and log out and SSH back in, or use newgrp docker.

5. Tools used by the clone script

command -v docker
docker compose version
command -v rsync

6. Primary (source) server

The core container mounts your SSH private key path and Docker socket so clone can run compose from the UI. SSH key path must exist on the host at the path you enter.

7. Host firewall agent (required for /firewall on standby)

Clone restores Postgres/images/env — not host listeners or UFW. The app talks to a small HTTP agent on the target host at FIREWALL_AGENT_URL (often http://<docker_bridge_gateway>:9191). If the agent is missing or UFW blocks that path, you see “Firewall agent unreachable”.

Recommended once per target after docker compose up succeeds:

cd /path/to/mtr-web-live
sudo apt-get install -y ufw curl openssl
bash scripts/security/bootstrap_firewall_agent.sh

Ensure SSH stays allowed before enabling UFW (sudo ufw allow OpenSSH etc.). For DR parity with primary’s rules through the UI, use Firewall after the agent is healthy; importing raw host UFW from primary is optional and machine-specific.

8. Fresh test on standby (optional — destructive)

On the failover machine only, to wipe the app folder and Docker volumes for a clean rerun (adjust path if not ~/mtr-web-live):

cd ~/mtr-web-live 2>/dev/null || cd /opt/mtr-web-live
docker compose down -v 2>/dev/null || true
cd ..
rm -rf ~/mtr-web-live /opt/mtr-web-live

Warning: docker compose down -v deletes named volumes (database data). Only run when you intend a full reset.
